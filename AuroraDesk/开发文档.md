# Windows 壁纸引擎开发文档

## 1. 项目目标

开发一个支持 **Windows** 的壁纸引擎（AuroraDesk），支持以下能力：

* 多显示器独立设置不同壁纸
* 支持静态图片壁纸
* 支持视频壁纸
* 支持 HTML 壁纸
* 用户可添加本地资源
* 有设置界面、预览、资源管理
* 具备基本性能控制
* 后续可扩展动态图片、互动壁纸、创意工坊式资源包等

---

## 2. 产品定位

### 2.1 首版目标

先做出一个 **稳定、可用、性能可控** 的 Windows 壁纸播放器，而不是一开始就做成 Wallpaper Engine 全量替代。

### 2.2 首版范围

首版建议只做：

* 图片壁纸
* 视频壁纸
* HTML 壁纸
* 多屏独立设置
* 本地资源导入
* 托盘驻留
* 开机自启
* 全屏时自动暂停
* 基础设置页

### 2.3 暂不做

这些功能先放到后期：

* 在线资源商店
* 社区/创意工坊
* 壁纸编辑器
* 音频响应特效
* 鼠标粒子/复杂交互特效
* 云同步
* 多平台支持

---

# 3. 推荐技术选型

## 3.1 主框架

### 推荐

**C# + WPF + .NET 8**

### 原因

* Windows 原生桌面开发成熟
* 与 Win32 API 互操作方便
* 做设置界面、资源管理、托盘程序效率高
* 适合先做稳定 MVP

---

## 3.2 核心依赖

### UI

* WPF

### HTML 渲染

* **WebView2**

### 视频播放

二选一：

* **LibVLCSharp**：更容易上手，适合首版
* **libmpv**：更强，但集成复杂度更高

建议首版先用 **LibVLCSharp**，后面再评估切到 mpv。

### Windows 底层

* Win32 API
* P/Invoke

### 数据存储

* SQLite 或 JSON
* 首版建议：

  * 配置用 JSON
  * 资源索引可以先用 SQLite，也可以先 JSON，后面再升级

### 缩略图/缓存

* 本地文件缓存目录
* 图片直接生成 thumbnail
* 视频提取首帧或固定时间点帧

# 4. 总体架构设计

---

## 4.1 系统分层

整个系统建议拆成下面几层：

### 1）UI 层

负责：

* 设置页
* 壁纸库
* 屏幕选择
* 导入资源
* 预览
* 托盘菜单
* 日志查看

### 2）应用服务层

负责：

* 配置读写
* 壁纸切换
* 屏幕映射
* 播放控制
* 暂停/恢复策略
* 启动项管理

### 3）壁纸运行时层

负责：

* 在桌面层创建宿主窗口
* 每个屏幕维护一个壁纸实例
* 生命周期管理
* 异常恢复

### 4）渲染层

不同类型用不同 Renderer：

* `ImageRenderer`
* `VideoRenderer`
* `HtmlRenderer`

### 5）基础设施层

负责：

* 文件导入
* 缩略图生成
* 缓存
* 日志
* 数据库/配置
* 资源扫描

---

## 4.2 关键概念模型

### MonitorProfile

表示一块显示器

字段示例：

* `MonitorId`
* `DeviceName`
* `Bounds`
* `WorkArea`
* `IsPrimary`
* `DpiX`
* `DpiY`

### WallpaperItem

表示一个壁纸资源

字段示例：

* `Id`
* `Type`（Image / Video / Html）
* `SourcePath`
* `Title`
* `ThumbnailPath`
* `Duration`
* `Width`
* `Height`
* `Tags`
* `AddedTime`

### WallpaperAssignment

表示某个屏幕当前绑定的壁纸

字段示例：

* `MonitorId`
* `WallpaperId`
* `PlaybackOptions`
* `ScaleMode`
* `Volume`
* `Mute`
* `PausePolicy`

### PlaybackOptions

字段示例：

* `Loop`
* `Mute`
* `Volume`
* `PlaybackRate`
* `FitMode`
* `PauseWhenFullscreen`
* `PauseWhenBatteryLow`

---

## 4.3 每屏独立宿主模型

不要做成“一个大窗口跨所有屏幕”。

正确做法是：

* 枚举所有显示器
* 每个显示器创建一个宿主窗口
* 每个窗口加载对应壁纸
* 每个窗口独立控制

这样后面做：

* 单屏不同壁纸
* 单屏暂停/恢复
* 单屏重载
* 单屏崩溃恢复

都会更简单。

# 阶段 1：静态图片壁纸 MVP

## 目标

先做最简单、最稳定的版本：**单屏图片壁纸**

## 为什么先做这个

这是最容易落地的功能，可以先把“桌面宿主 + 资源加载 + 壁纸显示”这条链跑通。

## 要做什么

### 1.1 桌面宿主窗口

实现一个能显示在桌面图标下层或合适桌面层的宿主窗口。

### 1.2 图片渲染器

实现 `ImageRenderer`：

* 加载本地图片
* 支持 jpg/png/webp
* 支持填充/适应/拉伸/居中
* 支持切换图片

### 1.3 最小设置界面

做一个简陋但可用的 UI：

* 选择本地图片
* 应用到当前主屏
* 查看当前状态

### 1.4 配置持久化

记录：

* 当前壁纸路径
* 显示模式
* 开机恢复

## 技术重点

* 宿主窗口大小与屏幕同步
* 窗口不抢焦点
* 图片加载和释放
* 配置保存

## 产出

* 第一版可运行产品
* 可设置主屏静态图片壁纸

## 验收标准

* 重启程序后可恢复之前的壁纸
* 图片显示正确，无明显闪烁
* 窗口不遮挡正常桌面使用

# 阶段 2：多屏幕支持（图片）

## 目标

支持多显示器独立设置不同图片壁纸。

## 为什么在这里做

因为多屏是核心卖点，但在图片阶段做最简单，问题最少。

## 要做什么

### 2.1 显示器枚举

读取所有显示器信息：

* 屏幕编号
* 分辨率
* 位置坐标
* 主副屏
* DPI

### 2.2 每屏独立宿主

为每个显示器创建一个宿主窗口。

### 2.3 每屏独立配置

实现：

* 屏幕 A 设置图片 1
* 屏幕 B 设置图片 2
* 配置持久化

### 2.4 屏幕热插拔处理

处理：

* 外接显示器插入
* 显示器移除
* 分辨率变化

## 技术重点

* 多显示器坐标系统
* 虚拟桌面坐标
* 每个宿主窗口与监视器的绑定
* 热插拔后的恢复策略

## 产出

* 支持多屏独立图片壁纸的版本

## 验收标准

* 两块以上显示器可分别设置不同图片
* 拔掉副屏后程序不崩溃
* 分辨率改变后壁纸自动适配

支持：

# 阶段 5：HTML 壁纸支持

## 目标

支持本地 HTML 壁纸。

## 为什么排在视频后

HTML 壁纸比图片复杂，也涉及 WebView2 生命周期、资源路径、权限等问题，但比互动特效和编辑器简单。

## 要做什么

### 5.1 接入 WebView2

在桌面宿主中加载 WebView2

### 5.2 HtmlRenderer

支持：

* 打开本地 HTML 文件
* 加载 HTML 目录入口文件
* 处理本地资源引用
* 刷新/重载
* 捕获加载失败

### 5.3 HTML 壁纸资源格式

建议约定结构：

```text
MyWallpaper/
├─ index.html
├─ css/
├─ js/
├─ assets/
└─ manifest.json
```

### 5.4 manifest 规范

定义基本字段：

* title
* author
* version
* preview
* entry
* description

### 5.5 HTML 壁纸导入

支持导入 zip 包或文件夹。

## 技术重点

* 本地文件权限
* WebView2 生命周期管理
* 加载失败回退
* JS 崩溃/卡死时的处理

## 产出

* HTML 壁纸支持
* 简单资源包格式规范

## 验收标准

* 本地 HTML 能显示
* 带 CSS/JS 资源的页面能正常加载
* 切换 HTML 壁纸不残留旧进程或旧页面

# 阶段 6：统一播放控制与运行时管理

## 目标

把图片、视频、HTML 的控制逻辑统一起来，避免后面系统越来越乱。

## 要做什么

### 6.1 定义统一接口

例如：

* `IWallpaperRenderer`
* `Initialize()`
* `Load()`
* `Play()`
* `Pause()`
* `Stop()`
* `Dispose()`

### 6.2 定义 WallpaperHost

宿主只负责：

* 窗口生命周期
* 显示器绑定
* 承载 Renderer

### 6.3 定义 RuntimeManager

负责：

* 监视器变化
* 资源切换
* 异常恢复
* 启停顺序

### 6.4 切换机制

壁纸切换流程统一：

* 停旧实例
* 释放资源
* 创建新实例
* 新实例就绪后显示

## 技术重点

* 统一状态机
* 资源释放时机
* 崩溃隔离
* 避免不同 Renderer 自己乱管窗口

## 产出

* 更清晰的运行时架构
* 更可维护的代码库

## 验收标准

* 三类壁纸都走统一切换流程
* 频繁切换资源不容易出 bug
* 新增 renderer 时改动范围可控

# 8. 每阶段的开发重点提醒

---

## 8.1 最容易踩坑的点

### 桌面挂载

你最核心、最早要搞定的问题不是 UI，而是：

* 窗口到底挂在哪一层
* 是否会挡住桌面图标
* Explorer 重启后是否丢失
* 多屏窗口位置是否正确

这个必须最早 PoC。

### 视频释放

视频播放器最常见问题：

* 切换资源时窗口黑屏
* 内存不释放
* 多次切换后崩溃
* 后台线程没退出

所以一定要尽早建立统一的生命周期接口。

### WebView2

HTML 壁纸会带来：

* 本地资源路径
* JS 死循环
* GPU 占用
* 页面崩溃
* WebView2 运行时依赖

所以不要在前两阶段就大量押宝 HTML。

# 13. 最终建议

如果你现在准备正式开工，最合理的落地路线是：

**先做图片，再做多屏，再做资源库，再做视频，再做 HTML，最后统一运行时和性能治理。**

不要反过来。
因为：

* 图片壁纸最容易验证桌面挂载
* 多屏图片最容易验证架构是否对
* 资源库先做，后面视频/HTML 才有统一入口
* 视频和 HTML 都会放大运行时管理问题
* 性能治理必须建立在功能已稳定的前提上
